## 🔄 GitHub × Obsidian × Cursor の統合運用

### 統合アーキテクチャ

```
┌─────────────────┐
│   Obsidian      │ 思考管理・アイデア・構造設計
│   (思考層)      │
└────────┬────────┘
         │
         │ 思考の転写
         ↓
┌─────────────────┐
│   Cursor        │ 実装・AI協創・自動化
│   (実装層)      │
└────────┬────────┘
         │
         │ 実装の管理
         ↓
┌─────────────────┐
│   GitHub        │ バージョン管理・Issue管理・公開
│   (管理層)      │
└─────────────────┘
```

---

### 運用フロー

#### 1. 思考段階（Obsidian）

**作業内容**:
- アイデアの記録
- 構造の設計
- 思考の整理

**ファイル形式**:
- Markdown（.md）
- Obsidian Vault構造

**同期**:
- ローカル保存
- 定期的なバックアップ

---

#### 2. 実装段階（Cursor）

**作業内容**:
- AI Orchestraとの協創
- コード実装
- ドキュメント作成

**ファイル形式**:
- コードファイル
- Markdown（.md）
- 設定ファイル

**同期**:
- Git管理
- GitHubへの自動プッシュ

---

#### 3. 管理段階（GitHub）

**作業内容**:
- バージョン管理
- Issue管理
- 公開・共有

**ファイル形式**:
- Gitリポジトリ
- GitHub Issues
- GitHub Actions

**同期**:
- Git push/pull
- GitHub Actionsによる自動化

---

### 統合の自動化

#### 1. Obsidian → Cursor

**自動化内容**:
- Obsidianの特定フォルダを監視
- 変更を検知したらCursorに通知
- 自動的に実装フォルダに転写

**実装方法**:
- ファイル監視スクリプト
- 自動転写スクリプト

---

#### 2. Cursor → GitHub

**自動化内容**:
- Cursorでの変更を自動的にGitにコミット
- 自動的にGitHubにプッシュ
- Issueの自動作成

**実装方法**:
- Git hooks
- GitHub Actions
- 自動コミットスクリプト

---

#### 3. GitHub → Obsidian

**自動化内容**:
- GitHub IssuesをObsidianに同期
- 完了したIssueをObsidianに記録
- プロジェクトの進捗をObsidianに反映

**実装方法**:
- GitHub API
- 同期スクリプト
- 自動更新スクリプト

## 📊 プロジェクトの優先順位付け基準

### 優先度の定義

#### P1: 最優先（Critical）

**基準**:
- 期限が迫っている
- 収益に直結する
- ブランドに重要な影響を与える

**対応**:
- 即座に対応
- 全リソースを投入
- 毎日進捗を確認

---

#### P2: 高優先度（High）

**基準**:
- 重要なプロジェクト
- 中期的な収益に影響
- ブランドに影響を与える

**対応**:
- 優先的に対応
- リソースを確保
- 週次で進捗を確認

---

#### P3: 中優先度（Medium）

**基準**:
- 通常のプロジェクト
- 長期的な収益に影響
- ブランドに間接的な影響

**対応**:
- 通常の対応
- リソースを適切に配分
- 月次で進捗を確認

---

#### P4: 低優先度（Low）

**基準**:
- 将来のプロジェクト
- 実験的なプロジェクト
- ブランドへの影響が小さい

**対応**:
- 余裕があるときに対応
- 最小限のリソース
- 四半期で進捗を確認

---

### 優先順位付けの自動化

**方法**:
- プロジェクトのメタデータに優先度を設定
- 自動的に優先順位を計算
- ダッシュボードで可視化

**基準**:
- 期限
- 収益への影響
- ブランドへの影響
- リソースの必要性
